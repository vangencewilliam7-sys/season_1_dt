# Season 1 Digital Twin: Detailed Architecture Document (HLA & LLA)

> **Overview**: A comprehensive architectural breakdown of the **Doctor Digital Twin**, a high-fidelity AI clone of a Fertility Specialist (or any domain expert). This system is designed to ground its behavior in verified clinical decisions, capturing not just facts, but the exact approach, tone, psychology, and logic of the expert. 

---

## 1. What is the Digital Twin?

The Digital Twin is not merely a chatbot or a search engine; it is an AI representation of an expert constructed through a strict provenance chain. Every output generated by the Digital Twin can be traced back to a specific source document, an expert transcript, or an explicit decision recorded in the system. The system degrades gracefully by switching to a "fallback mode" when confidence drops, preventing hallucination.

The architecture is divided into three major blocks across **five sequential acts**:
1. **Knowledge Hub** (Acts 1-3): Ingestion, structuring, embedding, and continuous learning from the expert's documents and decisions.
2. **Expert Persona** (Act 4): The extraction framework that translates raw knowledge into a living persona.
3. **Runtime Retrieval & Skills** (Act 5): The live execution layer that handles user queries and triggers actions.

---

## 2. High-Level Architecture (HLA)

The system connects the structured knowledge base (Knowledge Hub) to the AI persona generation (Expert Persona), exposing a runtime environment for user interactions.

```text
┌─────────────────────────────────────────────────────────────────────────┐
│                          KNOWLEDGE HUB                                   │
│  [Documents] → [Hierarchical Parser] → [Embeddings & RAG] → [DB Storage] │
│                                  ↓                                       │
│                    [Master Case Generation Loop]                         │
└──────────────────────────────────┬──────────────────────────────────────┘
                                   │ (SupabaseReader Bridge)
┌──────────────────────────────────▼──────────────────────────────────────┐
│                         EXPERT PERSONA                                   │
│  [Knowledge Extraction] → [AI Journalist Interview] → [Persona Manifest] │
│                                  ↓                                       │
│                       [Shadow Mode Approval]                             │
└──────────────────────────────────┬──────────────────────────────────────┘
                                   │ (Approved Manifest JSON)
┌──────────────────────────────────▼──────────────────────────────────────┐
│                    SKILLS & RUNTIME RETRIEVAL                           │
│  [User Query] → [Intent & RAG Router] → [Confidence Scoring]             │
│                                  ↓                                       │
│      (Primary Persona) ← [3-Layer Prompt Assembly] → (Fallback Persona)  │
│                                  ↓                                       │
│      [Functional Skills Execution] → [Response with Citations]           │
└─────────────────────────────────────────────────────────────────────────┘
```

### Core Design Principles (SOLID)
- **Single Responsibility:** Each module has exactly one job (e.g., KB structures knowledge, Persona extracts behavior, Skills perform actions).
- **Open/Closed:** Adding a new domain expert (e.g., `HealthcareAdapter`) or a new skill requires adding a new class, without modifying the core.
- **Liskov Substitution:** The system relies on drop-in replacements like `SupabaseReader` wherever `BaseDocumentReader` is expected.
- **Interface Segregation:** Distinct interfaces exist for LLM Providers, Document Readers, Domain Adapters, and Skills.
- **Dependency Inversion:** Modules depend on abstractions (e.g., `shared/interfaces/`), never on concrete implementations.

## 3. Low-Level Architecture (LLA)

The LLA provides a granular look at the mechanics within each of the five operational acts of the pipeline.

### ACT 1 — Knowledge Hub: Document Ingestion & Structure
**Goal:** Convert raw formats into normalized, gap-preserving structured data.
1. **Raw Ingestion:** Uploads of PDFs, DOCX, and transcripts are processed.
2. **Hierarchical Parser:** Structural headings are extracted (H1 → H2 → H3 → body).
3. **Universal Structure Builder:** Documents are mapped to a standardized hierarchy. If an H2 is missing between an H1 and H3, the parser inserts a placeholder gap to maintain the integrity of the knowledge skeleton.
4. **Output:** A list of `DocumentChunk` objects, where each chunk maintains its `parent_id`, `level`, `content`, and `source_path`.

### ACT 2 — Knowledge Hub: Metadata, RAG & Embeddings
**Goal:** Enable contextual follow-ups by preserving knowledge lineage.
1. **Metadata Assignment:** Metadata (timestamps, tags, path) is mapped to each `DocumentChunk`.
2. **Embedding:** The Embedding Service vectorizes every chunk (1536-dimensional using OpenAI's `text-embedding-3-small`).
3. **Vector Database:** Embeddings and structured chunks are stored in Supabase with `pgvector` to enable cosine similarity retrieval.

### ACT 3 — Knowledge Hub: Validation, Storage & Master Cases
**Goal:** Crystallize raw text into expert logic and decisions.
1. **Structural Audit:** Validates parent-child chains. Orphan nodes and tree breaks are flagged.
2. **Storage:** Stored in Supabase under UUID namespaces for each expert.
3. **Master Case Generation:** Scans all documents to generate `MasterCase` objects:
   - `expert_decision`: The actual decision.
   - `chain_of_thought`: How they reached the decision.
   - `logic_tags`: Contextual labels.
   - `confidence_note`: Certainty indicators.
4. **Implicit Knowledge Capture (Divergence Scanner):** Identifies soft-rule markers (e.g., "typically," "usually") to force the expert to clarify edges cases, transforming tacit knowledge into explicit data.

### ACT 4 — Expert Persona: Extraction & Manifestation
**Goal:** Synthesize the expert's approach, tone, and psychology into a `PersonaManifest`.
1. **Knowledge Extraction:** The `SupabaseReader` continuously ingests chunks and Master Cases. Redundancy is intentional here to ensure the latest expert state is captured.
2. **AI Journalist Pipeline:**
   - Evaluates KB/Master Cases to generate targeted, domain-specific interview questions mapped to gaps/hypotheses.
   - Analyzes expert responses (via transcript or text) to extract behavioral traits.
   - Employs an LLM fallback loop to predict plausible behaviors where gaps remain, which the expert later approves/corrects.
3. **Persona Manifestation:** The extracted traits are compiled into a `PersonaManifest` JSON:
   - `identity`: Name, Role, Domain.
   - `communication_style`: Tone, verbosity, and preferred framings.
   - `heuristics`: Array of `{trigger, decision, reasoning}` defining the 'WHY' behind their decisions.
   - `drop_zone_triggers`: Topics that the persona must rigidly refuse to handle.
   - `confidence_threshold`: The numerical confidence score required to answer as the expert.
4. **Shadow Mode:** The Manifest must be manually reviewed and shadow-approved before moving to production.

### ACT 5 — Runtime: Skills Execution & Retrieval Pipeline
**Goal:** Route user queries intelligently, retrieve cases, build the prompt, and execute required actions.

#### The Skill Architecture
The Digital Twin employs a two-tier skill system allowing for dynamic behavior:
1. **Base Skills (Infrastructure):** Raw, reusable capabilities such as `EmailSkill`, `MessageSkill`, `SearchKBSkill`.
2. **Functional Skills (Contextualized Execution):** Wraps Base Skills with the `PersonaManifest`. Example: `StudentNotificationSkill` (uses `EmailSkill` but writes in the exact tone of the expert and references the expert's Master Cases).

#### The Retrieval Pipeline
The runtime strictly enforces how answers are formulated:
1. **Query & Routing:** The user's query passes through an `Intent Router` which assesses the risk level.
2. **RAG Retrieval:** A `pgvector` cosine search looks for the most relevant Master Cases.
3. **Confidence Scorer:** 
   - If the `max(cosine_scores)` **≥** `confidence_threshold`, the system adopts the **Primary Persona** (answering AS the expert).
   - If the score **<** `confidence_threshold`, the system adopts the **Fallback Persona** (answering in a Deputy mode or escalating).
4. **3-Layer Prompt Assembly:** The prompt passed to the LLM is constructed rigidly:
   - **Layer 1: Domain Safety Rules** (Immutable constraints injected first).
   - **Layer 2: PersonaManifest JSON** (Identity, tone, heuristics).
   - **Layer 3: Retrieved Master Cases** (Specific RAG findings).
   - *Result*: An LLM response fully bounded by safety rules and grounded in the expert's explicit history.

## 4. Technical Data Flow

1. **Document Upload** → KB parses, structures, embeds, validates, and stores.
2. **Scan & Loop** → KB extracts Master Cases and initiates the Implicit Knowledge loop to catch tacit rules.
3. **Bridge Connection** → `BaseDocumentReader` reads KB chunks and Master Cases.
4. **Persona Generation** → Ingests data → AI Journalist interviews → Compiles findings → Generates `PersonaManifest.json`.
5. **Approval** → Shadow Mode HITL (Human-In-The-Loop) review.
6. **Runtime Retrieval** → Query arrives → RAG runs → Confidence scored → Routes to Persona/Deputy → Output response with source citations.
